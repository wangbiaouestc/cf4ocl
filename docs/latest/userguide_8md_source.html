8 # Overview {#ug_overview}
9 
10 The C Framework for OpenCL, _cf4ocl_, is a cross-platform pure C
11 object-oriented framework for developing and benchmarking [OpenCL](http://www.khronos.org/opencl/)
12 projects in C. It aims to:
13 
14 1. Promote the rapid development of OpenCL programs in C, with support
15 for C++.
16 2. Assist in the benchmarking of OpenCL events, such as kernel execution
17 and data transfers.
18 3. Simplify the analysis of the OpenCL environment and of kernel
19 requirements.
20 
21 ## Features {#ug_features}
22 
23 * Object-oriented interface to the OpenCL API
24  * New/destroy functions, no direct memory alloc/free
25  * Easy (and extensible) device selection
26  * Simple event dependency mechanism
27  * User-friendly error management
28 * OpenCL version independent
29 * Integrated profiling

35 ### Library organization {#ug_forg}
36 
37 The _cf4ocl_ library offers an object-oriented interface to the OpenCL
38 API using wrapper classes and methods (or structs and functions, in C
39 terms), grouped into modules of the same name, as shown in the following
40 table:
41 
42 | _cf4ocl_ module | _cf4ocl_ wrapper class | Wrapped OpenCL type |
43 | --------------------------------------- | ---------------------- | ------------------- |
44 | @ref CCL_PLATFORM_WRAPPER "Platform module" | ::CCLPlatform* | cl_platform_id |
45 | @ref CCL_DEVICE_WRAPPER "Device module" | ::CCLDevice* | cl_device_id |
46 | @ref CCL_CONTEXT_WRAPPER "Context module" | ::CCLContext* | cl_context |
47 | @ref CCL_QUEUE_WRAPPER "Queue module" | ::CCLQueue* | cl_command_queue |
48 | @ref CCL_PROGRAM_WRAPPER "Program module" | ::CCLProgram* | cl_program |
49 | @ref CCL_KERNEL_WRAPPER "Kernel module" | ::CCLKernel* | cl_kernel |
50 | @ref CCL_EVENT_WRAPPER "Event module" | ::CCLEvent* | cl_event |
51 | @ref CCL_MEMOBJ_WRAPPER "MemObj module" | ::CCLMemObj* | cl_mem |
52 | @ref CCL_BUFFER_WRAPPER "Buffer module" | ::CCLBuffer* | cl_mem |
53 | @ref CCL_IMAGE_WRAPPER "Image module" | ::CCLImage* | cl_mem |
54 | @ref CCL_SAMPLER_WRAPPER "Sampler module" | ::CCLSampler* | cl_sampler |
55 
56 Some of the provided methods directly wrap OpenCL functions (e.g.
57 ::ccl_buffer_enqueue_copy()), while others perform a number of OpenCL
58 operations in one function call
59 (e.g. ::ccl_kernel_set_args_and_enqueue_ndrange()). The wrapper classes
60 are organized in a hierarchical @ref ug_architecture "inheritance tree".
61 
62 Additional modules are also available:
63 
64 | _cf4ocl_ module | Description |
65 | ---------------------------------------------- | -------------------------------------------------------------------------------------------------- |
66 | @ref CCL_DEVICE_SELECTOR "Device selector module" | Automatically select devices using filters. |
67 | @ref CCL_DEVICE_QUERY "Device query module" | Helpers for querying device information, mainly used by the @ref ug_devinfo "ccl_devinfo" program. |
68 | @ref CCL_ERRORS "Errors module" | Convert OpenCL error codes into human-readable strings. |
69 | @ref CCL_PLATFORMS "Platforms module" | Management of the OpencL platforms available in the system. |
70 | @ref CCL_PROFILER "Profiler module" | Simple, convenient and thorough profiling of OpenCL events. |

72 ### The new/destroy rule {#ug_new_destroy}
73 
74 The _cf4ocl_ constructors and destructors have `new` and `destroy` in
75 their name, respectively. In _cf4ocl_, the new/destroy rule states
76 the following:
77 
78 _For each invoked constructor, the respective destructor must also be
79 invoked._
80 
81 This might seem obvious, but in many instances several objects are
82 obtained using other (non-constructor) methods during the course of a
83 program. These objects are automatically released and should not be
84 destroyed by client code.
85 
86 For example, it is possible to get a kernel belonging to a program
87 using the ::ccl_program_get_kernel() function:
88 
89 ~~~~~~~~~~~~~~~{.c}
90 CCLProgram* prg;
91 CCLKernel* krnl;
92 ~~~~~~~~~~~~~~~
93 ~~~~~~~~~~~~~~~{.c}
94 prg = ccl_program_new_from_source_file(ctx, "myprog.cl", NULL);
95 ~~~~~~~~~~~~~~~
96 ~~~~~~~~~~~~~~~{.c}
97 krnl = ccl_program_get_kernel(prg, "someKernel", NULL);
98 ~~~~~~~~~~~~~~~
99 
100 The returned kernel wrapper object will be freed when the program
101 is destroyed; as such, there is no need to free it. Destroying the program will
102 suffice:
103 
104 ~~~~~~~~~~~~~~~{.c}
105 ccl_program_destroy(prg);
106 ~~~~~~~~~~~~~~~

108 ### Getting info about OpenCL objects {#ug_getinfo}
109 
110 The `ccl_<class>_get_info_<scalar|array>()` macros can be used to get
111 information about OpenCL objects. Use the `array` version when the
112 expected return value is a pointer or array, or the `scalar` version
113 otherwise (e.g. when the expected return value is primitive or scalar
114 type).
115 
116 For example, to get the name and the number of compute cores on a
117 device:
118 
119 ~~~~~~~~~~~~~~~{.c}
120 CCLDevice* dev;
121 char* name;
122 cl_uint n_cores;
123 ~~~~~~~~~~~~~~~
124 ~~~~~~~~~~~~~~~{.c}
125 name = ccl_device_get_info_array(dev, CL_DEVICE_NAME, char*, NULL);
126 n_cores = ccl_device_get_info_scalar(dev, CL_DEVICE_MAX_COMPUTE_UNITS, cl_uint, NULL);
127 ~~~~~~~~~~~~~~~
128 
129 The `ccl_<class>_get_info()` macros serve more specific scenarios, and
130 are likely to be used less often. These macros return a
131 ::CCLWrapperInfo* object, which contains two public fields:
132 
133 * `value` - A pointer to the requested value as returned by the OpenCL
134 `clGet*Info()` functions.
135 * `size` - The size in bytes of the value pointed to by the `value`
136 field.
137 
138 To use the value, a cast should be performed on the `value` field to
139 convert it to the required type (which is what the
140 `ccl_<class>_get_info_<scalar|array>()` macros automatically do).
141 
142 The values and objects returned by these macros are automatically
143 released when the respective wrapper object is destroyed and should
144 never be directly freed by client code.

146 ### Error handling {#ug_errorhandle}
147 
148 Error-reporting _cf4ocl_ functions provide two methods for client-side
149 error handling:
150 
151 1. The return value.
152 2. [GError](https://developer.gnome.org/glib/stable/glib-Error-Reporting.html#GError)-based
153 error-reporting.
154 
155 The first method consists of analysing the return value of a function.
156 Error-throwing functions which return a pointer will return `NULL` if
157 an error occurs. The remaining error-reporting functions return
158 `CL_FALSE` if an error occurs (or `CL_TRUE` otherwise). Client code can
159 check for errors by looking for `NULL` or `CL_FALSE` return values,
160 depending on the function. This error handling method does not provide
161 additional information about the reported error. For example:
162 
163 ~~~~~~~~~~~~~~~{.c}
164 CCLContext* ctx;
165 CCLProgram* prg;
166 ~~~~~~~~~~~~~~~
167 ~~~~~~~~~~~~~~~{.c}
168 prg = ccl_program_new_from_source_file(ctx, "program.cl", NULL);
169 if (!prg) {
170  fprintf(stderr, "An error ocurred");
171  exit(-1);
172 }
173 ~~~~~~~~~~~~~~~
174 
175 The second method uses the [GLib Error Reporting](https://developer.gnome.org/glib/stable/glib-Error-Reporting.html)
176 approach and is more flexible. A `GError` object is initialized to
177 `NULL`, and a pointer to it is passed as the last argument to the
178 function being called. If the `GError` object is still `NULL` after the
179 function call, no error has occurred. Otherwise, an error occurred
180 and it is possible to get a user-friendly error message:
181 
182 ~~~~~~~~~~~~~~~{.c}
183 CCLContext* ctx;
184 CCLProgram* prg;
185 GError* err = NULL;
186 ~~~~~~~~~~~~~~~
187 ~~~~~~~~~~~~~~~{.c}
188 prg = ccl_program_new_from_source_file(ctx, "program.cl", &err);
189 if (err) {
190  fprintf(stderr, "%s", err->message);
191  exit(-1);
192 }
193 ~~~~~~~~~~~~~~~
194 
195 An error domain and error code are also available in the `GError`
196 object. The domain indicates the module the error-reporting function is
197 located in, while the code indicates the specific error that occurred.
198 Three kinds of domain can be returned by error-reporting _cf4ocl_
199 functions, each of them associated with distinct error codes:
200 
201 | Domain | Codes | Description |
202 | --------------- | ----------------------------------------------------------------- | ----------------------------------------------------- |
203 | ::CCL_ERROR | ::ccl_error_code enum | Error in _cf4ocl_ not related with external libraries |
204 | ::CCL_OCL_ERROR | [cl.h](http://www.khronos.org/registry/cl/api/2.0/cl.h) | Error in OpenCL function calls |
205 | A GLib domain | [GLib-module](https://developer.gnome.org/glib/stable/) dependent | Error in GLib function call (file open/save, etc) |
206 
207 For example, it is possible for client code to act on different OpenCL
208 errors:
209 
210 ~~~~~~~~~~~~~~~{.c}
211 CCLContext* ctx;
212 CCLBuffer* buf;
213 GError* err = NULL;
214 ~~~~~~~~~~~~~~~
215 ~~~~~~~~~~~~~~~{.c}
216 buf = ccl_buffer_new(ctx, flags, size, host_ptr, &err);
217 if (err) {
218  if (err->domain == CCL_OCL_ERROR) {
219  /* Check if it's OpenCL error. */
220  switch (err->code) {
221  /* Do different things depending on OpenCL error code. */
222  case CL_INVALID_VALUE:
223  /* Handle invalid values */
224  case CL_INVALID_BUFFER_SIZE:
225  /* Handle invalid buffer sizes */
226  case CL_INVALID_HOST_PTR:
227  /* Handle invalid host pointer */
228 
229  /* Handle other OpenCL errors */
230 
231  }
232  } else {
233  /* Handle other errors */
234  }
235 }
236 ~~~~~~~~~~~~~~~
237 
238 Finally, if client code wants to continue execution after an error was
239 caught, it is mandatory to use the [g_clear_error()](https://developer.gnome.org/glib/stable/glib-Error-Reporting.html#g-clear-error)
240 function to free the error object and reset its value to `NULL`. Not
241 doing so is a bug, especially if more error-reporting functions are to
242 be called moving forward. For example:
243 
244 ~~~~~~~~~~~~~~~{.c}
245 CCLContext* ctx;
246 CCLProgram* prg;
247 GError* err = NULL;
248 ~~~~~~~~~~~~~~~
249 ~~~~~~~~~~~~~~~{.c}
250 prg = ccl_program_new_from_source_file(ctx, "program.cl", &err);
251 if (err) {
252  /* Print the error message, but don't terminate program. */
253  fprintf(stderr, "%s", err->message);
254  g_clear_error(&err);
255 }
256 ~~~~~~~~~~~~~~~
257 
258 Even if the program terminates due to an error, the [g_clear_error()](https://developer.gnome.org/glib/stable/glib-Error-Reporting.html#g-clear-error)
259 function can be still be called to destroy the error object.

261 ### Log messages {#ug_log}
262 
263 _cf4ocl_ uses the [GLib message logging framework](https://developer.gnome.org/glib/stable/glib-Message-Logging.html)
264 to log messages and warnings. _cf4ocl_ log output is handled by
265 [GLib's default log handler](https://developer.gnome.org/glib/stable/glib-Message-Logging.html#g-log-default-handler),
266 which outputs warnings and messages to `stderr`. If client code wishes
267 to redirect this output, it can do so by specifying another
268 [log function](https://developer.gnome.org/glib/stable/glib-Message-Logging.html#GLogFunc)
269 for the `cf4ocl2` log domain with
270 [g_log_set_handler()](https://developer.gnome.org/glib/stable/glib-Message-Logging.html#g-log-set-handler).
271 For example:
272 
273 @code{.c}
274 /* Log function which outputs messages to a stream specified in user_data. */
275 void my_log_function(const gchar *log_domain, GLogLevelFlags log_level,
276  const gchar *message, gpointer user_data) {
277 
278  g_fprintf((FILE*) user_data, "[%s](%d)>%s\n",
279  log_domain, log_level, message);
280 
281 }
282 @endcode
283 @code{.c}
284 FILE* my_file;
285 @endcode
286 @code{.c}
287 /* Add log handler for all messages from cf4ocl. */
288 g_log_set_handler("cf4ocl2", G_LOG_LEVEL_MASK | G_LOG_FLAG_FATAL | G_LOG_FLAG_RECURSION,
289  my_log_function, my_file);
290 @endcode

292 ## Wrapper modules {#ug_wrappers}
293 
294 Each [OpenCL class](http://www.khronos.org/registry/cl/sdk/2.0/docs/man/xhtml/classDiagram.html)
295 is associated with a _cf4ocl_ module. At the most basic level, each
296 module offers a wrapper class and functions which wrap or map their
297 OpenCL equivalents. This, in itself, already simplifies working with
298 OpenCL in C, because the _cf4ocl_ wrapper classes internally manages
299 directly related objects, which may be created during the course of a
300 program. As such, client code just needs to follow the
301 @ref ug_new_destroy "new/destroy" rule for directly created objects,
302 thus not having to worry with memory allocation/deallocation of
303 intermediate objects.
304 
305 In most cases, however, each _cf4ocl_ module also provides methods for
306 other common and not-so-common OpenCL host code patterns, allowing the
307 programmer to avoid their verbosity and focus on OpenCL device code.
308 
309 All _cf4ocl_ wrapper classes extend the ::CCLWrapper* abstract
310 wrapper class. The properties and methods of this class, which are
311 concerned with reference counts, wrapping/unwrapping of OpenCL objects
312 and getting object information, are essentially of internal use by other
313 _cf4ocl_ classes. This functionality is also available to client code
314 which requires a more advanced integration with _cf4ocl_, as explained
315 in the @ref ug_cclwrapper "advanced" section.
316 
317 Several OpenCL objects, namely `cl_platform_id`, `cl_context` and
318 `cl_program`, have a direct relationship with a set of `cl_device_id`
319 objects. In order to map this relationship, _cf4ocl_ provides the
320 ::CCLDevContainer* class, which is an intermediate class between the
321 ::CCLWrapper* parent class and the ::CCLPlatform*, ::CCLContext* and
322 ::CCLProgram* wrappers. The ::CCLDevContainer* class implements
323 functionality for managing a set of ::CCLDevice* wrapper instances.
324 This functionality is exposed to client code through concrete wrapper
325 methods. For example, the ::CCLContext* class provides the
326 ::ccl_context_get_all_devices(), ::ccl_context_get_device() and
327 ::ccl_context_get_num_devices() methods for this purpose.

405 ## ccl_devinfo program {#ug_devinfo}
406 
407 The `ccl_devinfo` program prints information about the OpenCL platforms
408 and devices available on the system. By default, only basic device
409 information, such as type, vendor or OpenCL version, is shown, allowing
410 the user to quickly assess what platforms and devices are available.
411 
412 All the available device information can be shown with the `-a` or
413 `--all` options. The `-c <param>` option, or its long variant,
414 `--custom=<param>`, allows the user to specify a partial or complete
415 `cl_device_info` parameter name. All matching parameters will be
416 printed. This option can be used several times.
417 
418 The `-n` or `--notfound` flags will force the program to shown known
419 parameters even if the respective information is not found in the
420 device. The `-l` flag, or its long variant `--list`, tells `ccl_devinfo`
421 to list all known device information parameters.
422 
423 Additional information about each listed parameter can be shown with
424 the `-v` or `--verbose` flags.
425 
426 The `ccl_devinfo` utility can also narrow down information to specific
427 platforms and devices with the `-p` and `-d` options, respectively
428 (or their long variants, `--platform` and `--device`). These options
429 accept the platform index or the (platform-wise) device index to perform
430 the selection.

456 ## Wrapper architecture {#ug_architecture}
457 
458 The wrapper classes, which wrap OpenCL types, are implemented using an
459 object-oriented (OO) approach in C. While C does not directly provide
460 OO constructs, it is possible to implement features such as inheritance,
461 polymorphism or encapsulation. Using this approach, _cf4ocl_ is able to
462 offer a clean and logical class system, while being available in a form
463 (C) which can be directly or indirectly invoked from other programming
464 languages.
465 
466 Each _cf4ocl_ wrapper class is defined by a source (.c) file and a
467 header (.h) file. The former contains the private class properties and
468 the method implementations, while the later defines its public API. The
469 class body is implemented in the source file as a C `struct`; the header
470 file provides an opaque pointer to it, which is the public side of the
471 class from a client code perspective. Inheritance is implemented by
472 including a member representing the parent class `struct` in the body of
473 the child class `struct`. This requires the sharing of parent class
474 implementations. In order to keep these opaque, the respective `struct`
475 is defined in "private" header files which are not included in the
476 public API. This way, instances of the child class can be cast to its
477 parent type when required. The child class `struct` effectively extends
478 the parent class `struct`. An example of this approach can be shown with
479 the definitions of the abstract ::CCLWrapper* class and of the concrete
480 ::CCLEvent* class, which extends ::CCLWrapper*:
481 
482 <em>In @c %_ccl_abstract_wrapper.h (not part of public API):</em>
483 @code{.c}
484 /* Base class for all OpenCL wrappers. */
485 struct ccl_wrapper {
486 
487  /* The wrapped OpenCL object. */
488  void* cl_object;
489 
490  /* Information about the wrapped OpenCL object. */
491  CCLWrapperInfoTable* info;
492 
493  /* Reference count. */
494  int ref_count;
495 
496 };
497 @endcode
498 
499 _In ccl_common.h:_
500 
501 @code{.c}
502 /* Event wrapper class type declaration. */
503 typedef struct ccl_event CCLEvent;
504 @endcode
505 
506 _In ccl_event_wrapper.c:_
507 
508 @code{.c}
509 /* Event wrapper class, extends CCLWrapper */
510 struct ccl_event {
511 
512  /* Parent wrapper object. */
513  CCLWrapper base;
514 
515  /* Event name, for profiling purposes only. */
516  const char* name;
517 
518 };
519 @endcode
520 
521 Methods are implemented as functions which accept the object on which
522 they operate as the first parameter. When useful, function-like macros
523 are also used as object methods, such as the case of the
524 @ref ug_getinfo "info macros". Polymorphism is not used, as the so
525 called "abstract" methods are just functions which provide common
526 operations to concrete methods, named differently for each concrete
527 class. For example, the @c ccl_dev_container_get_device() abstract
528 method is called by the ::ccl_context_get_device(),
529 ::ccl_platform_get_device() and ::ccl_program_get_device() concrete
530 methods, for which it provides common functionality.
531 
532 The _cf4ocl_ class hierarchy is shown in the following inheritance
533 diagram:
534 
535 @dot
536 digraph cf4ocl {
537  rankdir=RL;
538  node [shape=record, fontname=Helvetica, fontsize=10];
539  wrapper [ label="CCLWrapper*" URL="@ref ccl_wrapper"];

541  ctx [ label="CCLContext*" URL="@ref ccl_context"];
542  prg [ label="CCLProgram*" URL="@ref ccl_program"];

544  memobj [ label="CCLMemObj*" URL="@ref ccl_memobj"];
545  buf [ label="CCLBuffer*" URL="@ref ccl_buffer"];
546  img [ label="CCLImage*" URL="@ref ccl_image"];
547  dev [ label="CCLDevice*" URL="@ref ccl_device"];
548  evt [ label="CCLEvent*" URL="@ref ccl_event"];

550  queue [ label="CCLQueue*" URL="@ref ccl_queue"];

567 ## The CCLWrapper base class {#ug_cclwrapper}
568 
569 The ::CCLWrapper* base class holds several low-level responsibilities
570 for wrapper objects:
571 
572 * Wrapping/unwrapping of OpenCL objects and maintaining a one-to-one
573 relationship between wrapped objects and wrapper objects
574 * Low-level memory management (allocation and deallocation)
575 * Reference counting
576 * Information handling (i.e., handling of data returned by the several
577 `clGet*Info()` OpenCL functions)
578 
579 Wrapper constructors create the OpenCL object to be wrapped, but
580 delegate memory allocation to the special `ccl_<class>_new_wrap()`
581 functions. These accept the OpenCL object, and in turn call the
582 @c ccl_wrapper_new() function, passing it not only the object, but also
583 the size in bytes of the wrapper to be created. The
584 @c ccl_wrapper_new() function allocates memory for the wrapper
585 (initializing this memory to zero), and keeps the OpenCL object
586 (wrapping it) in the created wrapper instance. For example, the
587 ::ccl_kernel_new() creates the `cl_kernel` object with the
588 clCreateKernel() OpenCL function, but then relies on the
589 ::ccl_kernel_new_wrap() function (and thus, on @c ccl_wrapper_new())
590 for allocation and initialization of the new ::CCLKernel* wrapper object
591 memory.
592 
593 The destruction of wrapper objects and respective memory deallocation
594 is performed in a similar fashion. Each wrapper class has its own
595 `ccl_<class>_destroy()` method, but this method delegates actual
596 object release to the "abstract" @c ccl_wrapper_unref() function. This
597 function accepts the wrapper to be destroyed, its size in bytes, and
598 two function pointers: the first, with prototype defined by
599 @c ccl_wrapper_release_fields(), is a wrapper specific function for
600 releasing internal wrapper objects, which the super class has no
601 knowledge of; the second is the OpenCL object destructor function, with
602 prototype defined by @c ccl_wrapper_release_cl_object(). Continuing on
603 the kernel example, the ::ccl_kernel_destroy() method delegates kernel
604 wrapper destruction to @c ccl_wrapper_unref(), passing it the kernel
605 wrapper object, its size (i.e. `sizeof(` ::CCLKernel `)`), the "private"
606 (static in C) @c ccl_kernel_release_fields() function for destroying
607 kernel internal objects, and the clReleaseKernel() OpenCL kernel
608 destructor function.
609 
610 As such, all _cf4ocl_ wrapper objects use a common memory allocation and
611 deallocation strategy, implemented in the ::CCLWrapper* super class.
612 
613 The `ccl_<class>_new_wrap()` special constructors respect the
614 @ref ug_new_destroy "new/destroy" rule. Wrappers created with their
615 special constructor must be released with the respective
616 `ccl_<class>_destroy()` function. This allows client code to create
617 OpenCL objects directly with OpenCL functions, and then wrap the objects
618 to take advantage of _cf4ocl_ functionality and features. The OpenCL
619 object can be retrieved from its wrapper at all times with the
620 respective `ccl_<class>_unwrap` method.
621 
622 If `ccl_<class>_new_wrap()` functions are passed an OpenCL object which
623 is already wrapped, a new wrapper will not be created. Instead, the
624 existing wrapper is returned, with its reference count increased by 1.
625 Thus, there is always a one-to-one relationship between wrapped OpenCL
626 objects and their respective wrappers. In reality, the
627 `ccl_<class>_destroy()` functions decreases the reference count of the
628 respective wrapper, only destroying it if the reference count reaches
629 zero. Client code can increase and decrease the reference count of a
630 wrapper object using the associated `ccl_<class>_ref()` and
631 `ccl_<class>_unref()` macros. The `ccl_<class>_ref()` macros call the
632 ::ccl_wrapper_ref() function, casting the wrapper to its base class
633 (::CCLWrapper*), while the `ccl_<class>_unref()` macros are just aliases
634 for the respective `ccl_<class>_destroy()` functions.
635 
636 The ::CCLWrapper* class maintains a static hash table which associates
637 OpenCL objects (keys) to _cf4ocl_ wrappers (values). Access to this
638 table is thread-safe and performed by the @c ccl_wrapper_new() and
639 @c ccl_wrapper_unref() functions.
640 
641 The management of OpenCL object information is also handled by the
642 ::CCLWrapper* class. The ::ccl_wrapper_get_info() method accepts
643 two wrapper objects, the first being the object to query; the second is
644 an auxiliary object required by some lower-level OpenCL info functions,
645 such clGetKernelWorkGroupInfo(), which requires a device object besides
646 the kernel object. ::ccl_wrapper_get_info() also requires a pointer to
647 a OpenCL `clGet*Info()` function (with type defined as
648 ::ccl_wrapper_info_fp) in order to perform the desired query.
649 ::ccl_wrapper_get_info() returns a ::CCLWrapperInfo* object, which
650 contains two public properties: the queried value and its size. To be
651 useful, the value must be cast to the correct type. The
652 ::ccl_wrapper_get_info_value() and ::ccl_wrapper_get_info_size() methods
653 call ::ccl_wrapper_get_info(), but directly return the value and size of
654 the ::CCLWrapper* object, respectively.
655 
656 The requested information is kept in the information table of the
657 respective wrapper object. When the wrapper object is destroyed, all the
658 information objects are also released. As such, client code does not
659 need to worry about freeing objects returned by the
660 `ccl_wrapper_get_info*()` methods. These also accept a `use_cache`
661 boolean argument, which if true, causes the methods to first search
662 for the information in the wrappers information table, in case it has
663 already been requested; if not, they proceed with the query as normal.
664 
665 Client code will usually use the @ref ug_getinfo "info macros" of each
666 wrapper in order to fetch information about the underlying OpenCL
667 object. These macros expand into the `ccl_wrapper_get_info*()` methods,
668 automatically casting objects and values to the appropriate type,
669 selecting the correct `clGet*Info()` function for the object being
670 queried. The cache is never used by the @ref ug_getinfo "info macros".
671 
672 ## The CCLDevContainer class {#ug_ccldevcontainer}
673 
674 The intermediate ::CCLDevContainer* class provides functionality for
675 managing a set of ::CCLDevice* wrapper instances, abstracting code
676 common to the ::CCLPlatform*, ::CCLContext* and ::CCLProgram* classes,
677 all of which internally keep a set of devices. The ::CCLDevContainer*
678 class contains three "abstract" methods for accessing the associated set
679 of ::CCLDevice* wrappers, namely:
680 
681 * @c ccl_dev_container_get_all_devices() : get all ::CCLDevice* wrappers
682 in device container object.
683 * @c ccl_dev_container_get_device() : get ::CCLDevice* wrapper at given
684 index.
685 * @c ccl_dev_container_get_num_devices() : return number of devices in
686 device container object.
687 
688 Concrete wrapper implementations rely on this functionality, which
689 is exposed to client code via specific methods methods, e.g. in the case
690 of ::CCLProgram* objects, these are ::ccl_program_get_all_devices(),
691 ::ccl_program_get_device() and ::ccl_program_get_num_devices(),
692 respectively.

694 ## The CCLMemObj class {#ug_cclmemobj}
695

697 ::CCLImage* classes follows that of the respective [OpenCL types](http://www.khronos.org/registry/cl/sdk/1.2/docs/man/xhtml/classDiagram.html).
