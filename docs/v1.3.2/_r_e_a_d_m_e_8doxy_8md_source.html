1 Summary
2 =======
3 
4 The C Framework for OpenCL, _cf4ocl_, is a cross-platform pure C99 set
5 of libraries and utilities with the following goals:
6 
7 1. Promote the rapid development of OpenCL C programs.
8 2. Assist in the benchmarking of OpenCL events, such as kernel execution
9 and data transfers.
10 3. Simplify the analysis of the OpenCL environment and of kernel
11 requirements.
12 
13 _cf4ocl_ is divided into four parts: i) library ii) command-line
14 utilities iii) unit tests and, iv) examples. Part i), the library,
15 offers functions which aim to achieve goals 1 and 2. The command-line
16 utilities, part ii), are focused on goal 3. The unit tests, part
17 iii), aim to verify that the library functions are working properly.
18 Finally, the examples show how to integrate the library functions in
19 fully working OpenCL programs.
20 
21 _cf4ocl_ works in Linux, Windows and Mac OSX, although, being pure
22 C99, it should compile on other platforms, as long as the dependencies,
23 [GLib][] and [OpenCL][], are met. The reference compiler is GCC with
24 `-Wall` and `-Wextra` flags activated. The code is verified with
25 [cppcheck][] and is fully commented. API documentation can be generated
26 with [Doxygen][] via `make`.
27 
28 Library code is licensed under [LGPLv3][], while the remaining code is
29 licensed under [GPLv3][].

62 1\. Introduction
63 ============
64 
65 _cf4ocl_ is divided into four parts, with the following structure:
66 
67 1. Library
68  * CL Utils
69  * CL Profiler
70  * CL Errors
71  * GError framework
72 2. Library tests
73  * CL Profiler
74  * GError framework
75 3. Utilities
76  * Device query
77  * Kernel info
78 4. Examples
79  * Bank conflicts test
80  * Matrix multiplication
81 
82 The following sections describe each of the parts in additional detail.
83 
84 <a name="library"></a>
85 
86 1.1\. Library
87 -------
88 
89 The library offers functions which promote the rapid development of
90 OpenCL C programs, _CL Utils_, as well as their detailed benchmarking,
91 _CL Profiler_. The library also facilitates error management in two
92 ways: i) by converting OpenCL error codes into human readable strings,
93 _CL Errors_ ii) by exposing a generic error handling framework,
94 _GError framework_, used internally by _CL Utils_ and _CL Profiler_.
95 
96 <a name="clutils"></a>
97 
98 ### 1.1.1\. CL Utils
99 
100 The _CL Utils_ section of the library allows the initialization of
101 OpenCL environment objects, such as platform, device, context,
102 command queues and devices, with a single function, `clu_zone_new`, thus
103 avoiding the typical boilerplate code required for this setup. Device
104 selection is decoupled from the OpenCL environment setup, being
105 performed with a function passed as a parameter to `clu_zone_new`. The
106 library supplies functions to accomplish this task automatically or
107 through user interaction, although the client programmer can supply its
108 own function with the required selection criteria. The destruction of
109 the OpenCL environment, i.e. the removal of the respective objects from
110 memory, is also achieved with one function, namely `clu_zone_free`.
111 
112 Kernel compilation is simplified by the `clu_program_create` function,
113 which can be used to build OpenCL device programs by passing an array of
114 filenames containing kernel code.
115 
116 The data structures in _CL Utils_ are completely transparent to the
117 client programmer, allowing access the underlying OpenCL objects at all
118 times.
119 
120 <a name="clprofiler"></a>
121 
122 ### 1.1.2\. CL Profiler
123 
124 The functions in _CL Profiler_ allow the client programmer to obtain
125 detailed profiling information about OpenCL functions for kernel
126 execution and host-device memory transfers, including relative and
127 absolute execution times. Profiling information is obtained using
128 OpenCL events, which can be associated with the relevant OpenCL
129 functions. _CL Profiler_ supports multiple command queues and
130 overlapping events, such as simultaneous kernel executions and data
131 transfers. Profiling information can be exported in a configurable
132 format, and plotted using a script included in _cf4ocl_.
133 
134 <a name="clerrors"></a>
135 
136 ### 1.1.3\. CL Errors
137 
138 _CL Errors_ offers a single function which converts a OpenCL error into
139 a human readable string.
140 
141 <a name="gerrorframework"></a>
142 
143 ### 1.1.4\. GError framework
144 
145 The _GError framework_ is internally used by _CL Utils_ and
146 _CL Profiler_ for error handling purposes. However, the framework is not
147 in any way tied to OpenCL, so it can be used generically in any C
148 program. It is composed of three macros and it is based on the
149 `GError` object from GLib.

171 2\. How to use _cf4ocl_
172 ===================
173 
174 <a name="dependencies"></a>
175 
176 2.1\. Dependencies
177 ------------
178 
179 In order to compile _cf4ocl_ and use it in projects, two dependencies are
180 required: 1) [OpenCL][] and, 2) [GLib][] (v2.32 or newer).
181 
182 <a name="opencl"></a>
183 
184 ### 2.1.1\. OpenCL
185 
186 The OpenCL headers (for compiling) and shared library (for linking) can
187 be installed from the packages offered by different OpenCL vendors such
188 as [AMD][AMDSDK], [Intel][IntelSDK] or [Nvidia][NvidiaSDK].
189 
190 Alternatively, it is possible to install an OpenCL development
191 environment in a some recent Linux distributions. For example, in Debian
192 7 and Ubuntu 13.04:
193 
194  $ sudo apt-get install ocl-icd-opencl-dev
195 
196 This will install the headers and shared library system-wide, avoiding
197 the need to specify their location to GCC. However, to actually run an
198 OpenCL program it is necessary to have a vendor implementation
199 installed.
200 
201 <a name="glib"></a>
202 
203 ### 2.1.2\. GLib
204 
205 The simplest way to install [GLib][] on Linux is through the package
206 manager of the respective distribution. For Debian and Ubuntu:
207 
208  $ sudo apt-get install libglib2.0-dev
209 
210 It is also advisable to install the `libglib2.0-0-dbg` or equivalent
211 package to allow debugging at the GLib level.
212 
213 This [tutorial](http://www.gtk.org/download/win32_tutorial.php) explains
214 how to install GLib in Windows.
215 
216 <a name="downloadbuildandinstall"></a>
217 
218 2.2\. Download, build and install
219 ---------------------------
220 
221 _cf4ocl_ uses the [CMake][] build system. Assuming a working development
222 environment with GCC and CMake installed, and the dependencies (OpenCL
223 and GLib) properly setup, either clone or download and extract _cf4ocl_
224 into a location of your preference. The _cf4ocl_ directory tree will
225 have the following structure:
226 
227  cf4ocl/
228  |
229  +-- cmake
230  | |
231  | +-- Modules
232  |
233  +-- images
234  |
235  +-- scripts
236  |
237  +-- src
238  |
239  +-- examples
240  |
241  +-- lib
242  |
243  +-- tests
244  |
245  +-- utils
246 
247 Within the _cf4ocl_ directory, create a folder named `build`. This is
248 where the project will be built, such that the contents of the _cf4ocl_
249 folder (source code, scripts, etc.), are kept separate from the build
250 files.
251 
252 Next, `cd` into the `build` folder and run the `cmake ..` command. If
253 OpenCL and GLib are successfully found by CMake, the Makefiles required
254 for building and installing _cf4ocl_ are created. If an error occurrs,
255 it might be necessary to manually provide the location of these
256 dependencies. In order to do so, run the `ccmake ..` command, which
257 offers a configuration interface for this purpose. Alternatively, run
258 the CMake GUI, chose the _cf4ocl_ directory as the source code
259 location and the `build` directory as the place where to build the
260 binaries. Specify the library and include locations for OpenCL and
261 GLib, click "Configure", then "Generate". The makefiles should now have
262 been generated.
263 
264 In order to build the project, run `make` in the `build` folder.
265 Project installation can be accomplished with `make install`. By
266 default, _cf4ocl_ is installed system wide, which requires the later
267 command to be executed as root or administrator (e.g. using `sudo` in
268 Linux, or an elevated priviledge command prompt on Windows). The
269 install location can be changed with the `ccmake` command, or with the
270 CMake GUI. The tests can be compiled separately with `make tests`.
271 
272 After installation, the _cf4ocl_ shared library should be registered
273 with `ldconfig` (Linux) or `regsvr32` (Windows), such that client
274 applications can find it at runtime.
275 
276 The API documentation can be generated with `make` if [Doxygen][]
277 (version 1.8.3 or newer) is installed. In the _cf4ocl_ directory run:
278 
279  $ make doc
280 
281 The documentation will be generated in the `./doc` folder. The direct
282 use of the `doxygen` command should be avoided because
283 [Doxygen's markdown][doxymd] is not 100% compatible with the
284 [GitHub Flavored Markdown][ghmd] used in this document. When building
285 the documentation with `make`, the required conversion is performed
286 first.

290 2.3\. Using the library in a new project
291 ----------------------------------
292 
293 Any of the three elements of the library can be used in a workflow
294 without the need to use the others. However, both CL Utils and CL
295 Profiler internally use the GError framework. The _cf4ocl_ header
296 includes all the definitions for the three elements of the library. This
297 header can be included as follows:
298 
299 ~~~~~~~~~~~~~~~{.c}
300 #include <cf4ocl.h>
301 ~~~~~~~~~~~~~~~

358 
359 ### 2.3.2\. Using CL Utils
360 
361 The functions provided by CL Utils aim to facilitate the rapid
362 development and deployment of C programs which use OpenCL. These
363 functions are focused on two key tasks:
364 
365 1. Setup and tear down of the OpenCL environment, including device,
366 context and command queues.
367 
368 2. Loading, compilation and tear down of OpenCL kernel programs.
369 
370 These two tasks require very verbose code, with significant repetition
371 between projects. CL Utils wraps these tasks into two functions,
372 providing an additional third function for freeing up the allocated
373 resources. The OpenCL objects created by CL Utils are available to
374 the client programmer at all times through the `CLUZone` object.
375 CL Utils does not wrap any other OpenCL tasks, such as kernel execution
376 or data transfers, which can be performed directly with the respective
377 OpenCL functions.
378 
379 The first task is performed with the `clu_zone_new` function, which
380 has the following prototype:
381 
382 ~~~~~~~~~~~~~~~{.c}
383 CLUZone* clu_zone_new(
384  /* Type of device, e.g. CL_DEVICE_TYPE_CPU, CL_DEVICE_TYPE_GPU, etc. */
385  cl_uint deviceType,
386  /* Number of command queues. */
387  cl_uint numQueues,
388  /* Command queue properties, e.g. CL_QUEUE_PROFILING_ENABLE, etc. */
389  cl_int queueProperties,
390  /* Pointer to a device selector function. */
391  clu_device_selector devSel,
392  /* Extra arguments for device selector function. */
393  void* dsExtraArg,
394  /* GLib's error reporting object (may be ignored). */
395  GError **err
396 )
397 ~~~~~~~~~~~~~~~
398 
399 The returned `CLUZone` object will contain the context, command queues,
400 device information and an uninitialized kernel program. The program is
401 initialized with the `clu_program_create` function as part of the second
402 task. This function has the following prototype:
403 
404 ~~~~~~~~~~~~~~~{.c}
405 cl_int clu_program_create(
406  /* The CLUZone object return by the clu_zone_new function. */
407  CLUZone* zone,
408  /* List of files containing the source code for the kernels. */
409  const char** kernelFiles,
410  /* Number of kernel files given in the previous parameter. */
411  cl_uint numKernelFiles,
412  /* Compiler options. */
413  const char* compilerOpts,
414  /* GLib's error reporting object (may be ignored). */
415  GError **err
416 )
417 ~~~~~~~~~~~~~~~
418 
419 A typical usage pattern of CL Utils in a C program:
420 
421 ~~~~~~~~~~~~~~~{.c}
422 include "clutils.h"
423 
424 ...
425 
426 int main(int argc, char* argv[])
427 {
428  ...
429  CLUZone* zone = NULL
430  const char* oclSources[] = {"src1.cl", "src2.cl", "moresrc.cl"}
431  ...
432 
433  /* Get a complete OpenCL environment using a GPU device, one */
434  /* command queue, no command queue options, if more than one */
435  /* GPU is available show selection menu to user, and ignore */
436  /* error reporting. */
437  zone = clu_zone_new(CL_DEVICE_TYPE_GPU, 1, 0, clu_menu_device_selector, NULL, NULL)
438  ...
439 
440  /* Create (compile) an OpenCL kernel program using three OpenCL */
441  /* source files, no compiler options, and ignoring error */
442  /* reporting. */
443  status = clu_program_create(zone, oclSources, 3, NULL, NULL)
444  ...
445 
446  /* Perform OpenCL tasks such as data transfer and kernel execution */
447  /* using the OpenCL objects (queues, program, context, etc.) */
448  /* available within the CLUZone object. */
449  ...
450 
451  /* Free the CLUZone object and all of the contained OpenCL objects.*/
452  /* This object should only be released after all OpenCL objects */
453  /* not controlled by CL Utils (kernels, events, memory objects) */
454  /* are released. */
455  clu_zone_free(zone)
456 
457  ...
458 
459 }
460 ~~~~~~~~~~~~~~~
461 
462 The fourth parameter of `clu_zone_new` is a pointer to a device
463 selector function, which has the following definition:
464 
465 ~~~~~~~~~~~~~~~{.c}
466 typedef cl_uint (*clu_device_selector)(
467  /* Array of available devices and respective information. */
468  CLUDeviceInfo* devInfos,
469  /* Number of devices in array. */
470  cl_uint numDevices,
471  /* Extra arguments to selector function. */
472  void* extraArg
473 )
474 ~~~~~~~~~~~~~~~
475 
476 CL Utils provides two ready to use device selector functions:
477 
478 * `clu_menu_device_selector` - If more than one device exists, presents
479 a selection menu to the user. If extra argument is given it should be
480 the array index of the preferred device.
481 * `clu_info_device_selector` - Selects a device by device name, device
482 vendor and/or platform name, any/all of which are given in the extra
483 argument.
484 
485 For other device selection requirements, the client programmer can
486 develop a specific device selector function.
487 
488 <a name="usingclprofiler"></a>
489 
490 ### 2.3.3\. Using CL Profiler
491 
492 The goal of CL Profiler is to provide detailed benchmarking information
493 about OpenCL events such as kernel execution, data transfers, and so on.
494 CL Profiler is prepared to handle overlapping events, which usually
495 take place when the programmer is trying to optimize its application
496 by simultaneously transfer data to and from the OpenCL device and
497 execute kernels, using different command queues.
498 
499 CL Profiler consists of two files, `clprofiler.c` and `clprofiler.h`.
500 In order to use CL Profiler in a project, it is necessary to include
501 the `clprofiler.h` header file.
502 
503 For the purpose of this explanation, we will consider that two command
504 queues are being used:
505 
506 ~~~~~~~~~~~~~~~{.c}
507 cl_command_queue queue0 /* Used for host-device data transfer. */
508 cl_command_queue queue1 /* Used for kernel execution. */
509 ~~~~~~~~~~~~~~~
510 
511 Additionally, we will consider the following OpenCL events:
512 
513 ~~~~~~~~~~~~~~~{.c}
514 cl_event ev_transf_in /* Transfer data from host to device. */
515 cl_event ev_kernel_A_1 /* Execute kernel A on device. */
516 cl_event ev_kernel_B /* Execute kernel B on device. */
517 cl_event ev_kernel_A_2 /* Execute kernel A on device again. */
518 cl_event ev_transf_out /* Transfer data from device to host. */
519 ~~~~~~~~~~~~~~~
520 
521 The `ProfCLProfile` structure forms the basis of CL Profiler. It can:
522 1) measure the total elapsed time of the application (or the relevant
523 part of the application) and, 2) keep track of the device time required
524 by the OpenCL events. The following instruction creates a new
525 `ProfCLProfile` structure:
526 
527 ~~~~~~~~~~~~~~~{.c}
528 /* Create a new ProfCLProfile structure. */
529 ProfCLProfile* profile = profcl_profile_new()
530 ~~~~~~~~~~~~~~~
531 
532 In order to start measuring the relevant part of the application, the
533 following instruction should be issued:
534 
535 ~~~~~~~~~~~~~~~{.c}
536 /* Start basic timming / profiling. */
537 profcl_profile_start(profile)
538 ~~~~~~~~~~~~~~~
539 
540 At this time, the typical OpenCL application workflow, such as
541 transferring data and executing kernels, should take place. The above
542 defined events must be associated with the respective `clEnqueue*`
543 OpenCL functions in order to be later analyzed. A typical workflow
544 may be finalized with the following instructions:
545 
546 ~~~~~~~~~~~~~~~{.c}
547 /* Finish all pending OpenCL operations. */
548 clFinish(queue0)
549 clFinish(queue1)
550 ~~~~~~~~~~~~~~~
551 
552 Profiling should be stopped at this point.
553 
554 ~~~~~~~~~~~~~~~{.c}
555 profcl_profile_stop(profile)
556 ~~~~~~~~~~~~~~~
557 
558 Now the events can be added to the profiler structure. The
559 `profcl_profile_add` function uses the second parameter (a string)
560 as a key to differentiate between events. Thus, if the same key
561 is given for different OpenCL events, CL Profiler will consider
562 it to be the same _semantic_ event. This can be useful for aggregating
563 execution times of events which occur innumerous times in a cyclic
564 fashion (in a _for_ loop, for example).
565 
566 ~~~~~~~~~~~~~~~{.c}
567 /* Add events to be profiled/analyzed. */
568 profcl_profile_add(profile, "Transfer data to device", ev_transf_in, NULL)
569 profcl_profile_add(profile, "Kernel A", ev_kernel_A_1, NULL)
570 profcl_profile_add(profile, "Kernel B", ev_kernel_B, NULL)
571 profcl_profile_add(profile, "Kernel A", ev_kernel_A_2, NULL)
572 profcl_profile_add(profile, "Transfer data from device", ev_transf_out, NULL)
573 ~~~~~~~~~~~~~~~
574 
575 The above code will consider OpenCL events `ev_kernel_A_1` and `ev_kernel_A_2`
576 to be the same semantic event because the same key, string _Kernel A_, is
577 used. Thus, the total execution time for the semantic event _Kernel A_ will
578 be the sum of respective two OpenCL events. CL Profiler can even
579 determine overlaps of a semantic event with itself (i.e., two overlapping
580 OpenCL events which are added for analysis with the same key).
581 
582 After all the events are added, it is necessary to instruct CL
583 Profiler to perform the required calculations in order to determine
584 the absolute and relative times of all events, and how these correlate
585 with the total elapsed time of the relevant part of the application.
586 
587 ~~~~~~~~~~~~~~~{.c}
588 profcl_profile_aggregate(profile, NULL)
589 ~~~~~~~~~~~~~~~
590 
591 Finally, the complete benchmarking info can be printed:
592 
593 ~~~~~~~~~~~~~~~{.c}
594 profcl_print_info(profile, PROFCL_AGGEVDATA_SORT_TIME, NULL)
595 ~~~~~~~~~~~~~~~
596 
597 Two detailed tables will be shown on screen: one for individual events
598 (sorted by name or execution time), and another showing event overlaps,
599 if any occurred.
600 
601 A list of all events can be exported to a file or stream using the
602 `profcl_export_info*` functions.
603 
604 ~~~~~~~~~~~~~~~{.c}
605 profcl_export_info_file(profile, "profileinfo.txt", NULL)
606 ~~~~~~~~~~~~~~~
607 Export options, such as field delimiters, field separators, and so on,
608 can be set using the `profcl_export_opts_set()` function.
609 
610 A Gantt chart type graph can be produced from the exported file (using
611 the default options) via a Python script included with _cf4ocl_:
612 
613 `$ python plot_events.py profileinfo.txt`
614 
615 ![Events plot for file exported by test_profiler](images/gantt.png)
616 
617 A similar plot can also be produced with [gnuplot] using the [gantt.py]
618 script.
619 
620 The OpenCL events can be freed after they have been added to the
621 profiler structure and before the program terminates.
622 
623 ~~~~~~~~~~~~~~~{.c}
624 clReleaseEvent(ev_transf_in)
625 clReleaseEvent(ev_kernel_A_1)
626 clReleaseEvent(ev_kernel_B)
627 clReleaseEvent(ev_kernel_A_2)
628 clReleaseEvent(ev_transf_out)
629 ~~~~~~~~~~~~~~~
630 
631 The last parameter of some of the `profcl_*` functions is used for error
632 handling and to obtain detailed error messages if an error occurs. By
633 passing `NULL`, the programmer choses to ignore that feature. Such approach
634 is not critical because all of the error-prone `profcl_*` functions also return
635 their execution status code. No error handling is performed in this
636 explanation, though.
637 
638 For map events, CL Profiler provides the `profcl_profile_add_composite()`
639 function, which accepts two OpenCL events, one relative to the _map_
640 operation, and the other to the _unmap_ operation. The function uses
641 the start instant of the _map_ event, and the _end_ instant of the
642 _unmap_ event, in order to build a composite semantic event.
643 
644 <a name="usingclerrors"></a>
645 
646 ### 2.3.4\. Using CL Errors
647 
648 OpenCL functions usually return an error or status code, so that the
649 client programmer can check if the function call was successful, and if
650 not, what kind of problem occurred. The programmer can either check for
651 all possible errors for a given function call using the OpenCL error
652 constants (which map the error codes), or can just return or print the
653 error code, and then look at the [cl.h] header to determine what error
654 occurred. _CL Errors_ allows for a third option: convert the error code
655 into a human readable string. The usage is very simple, consisting of
656 a single call to the `clerror_get()` function, passing the error code
657 as an argument:
658 
659 ~~~~~~~~~~~~~~~{.c}
660 cl_int status
661 ...
662 status = clFinish(some_command_queue)
663 if (status != CL_SUCCESS) {
664  printf("An error occurred: %s\n", clerror_get(status))
665  ...
666 }
667 ...
668 ~~~~~~~~~~~~~~~
669 <a name="usinggerrorframeworkgerrorf"></a>
670 
671 ### 2.3.5\. Using GError Framework (GErrorF)
672 
673 The main purpose of GErrorF is to provide error handling constructs to
674 CL Utils and CL Profiler. No knowledge of GErrorF is required to use
675 CL Utils and/or CL Profiler. However, because it is sufficiently generic
676 to be used in any C application, a specific description is warranted.
677 
678 GErrorF uses GLib's `GError` object for function error reporting. This
679 explanation assumes some familiarity with
680 [GLib's error reporting][gliberror].
681 
682 GErrorF is defined by three macros in `gerrorf.h`:
683 
684 * `gef_if_error_create_goto` - Catches errors from non-GError aware
685 functions.
686 * `gef_if_error_goto` - Catches errors from GError aware functions.
687 * `gef_if_error_propagate_goto` - Same as previous macro, but propagates
688 error in a source GError object to a new destination GError object.
689 Useful for wrapping errors thrown by library / third-party GError aware
690 functions.
691 
692 GErrorF establishes an error handling methodology for C programs not
693 entirely dissimilar to the pattern used in Linux kernel development.
694 Any function producing recoverable runtime errors, from `main` to
695 functions located deeper in the call stack, can benefit from this
696 approach. The general usage of GErrorF is as follows:
697 
698 ~~~~~~~~~~~~~~~{.c}
699 include "gerrorf.h"
700 
701 ...
702 
703 int main(int argc, char* argv[])
704 {
705 
706  ...
707 
708  /* Must initialize every allocable pointers and objects */
709  /* to NULL. */
710  int some_vector* = NULL
711 
712  /* GError object. */
713  GError *err = NULL
714 
715  /* Function return status. */
716  int status = SUCCESS_CODE
717 
718  ...
719 
720  /* Call a GError aware function (defined bellow). */
721  some_function(params, &err)
722 
723  /* Catch possible error in GError-aware function. In this */
724  /* case the GError object is initialized by the called */
725  /* function. */
726  gef_if_error_goto(
727  err, /* GError object. */
728  SOME_ERROR_CODE, /* Error code to set in status. */
729  status, /* Function return status. */
730  error_handler /* Label to goto in case of error. */
731  )
732 
733  /* In the previous function it is possible to replace an app */
734  /* specific error code with GErrorF special constants */
735  /* GEF_USE_STATUS and GEF_USE_GERROR. The former leaves the */
736  /* status variable untouched (useful for cases where the */
737  /* function itself returns a usable int status), while the */
738  /* later sets status to the error code set in the GError */
739  /* object. */
740 
741  ...
742 
743  /* Call a non-GError aware function. */
744  some_vector = (int*) malloc(sizeof(int) * SOME_SIZE)
745 
746  /* Catch possible error in non-GError aware function. */
747  gef_if_error_create_goto(
748  err, /* GError object. */
749  SOME_QUARK_ERROR, /* GLib GQuark identifier. */
750  some_vector == NULL, /* Error condition. */
751  SOME_ERROR_CODE, /* Error code to set in err. */
752  error_handler, /* Label to goto in case of error. */
753  "Unable to alloc. mem." /* Error msg to set in err. */
754  )
755 
756  ...
757 
758  /* If we get here, there was no error, goto cleanup. */
759  g_assert(err == NULL) /* Make sure err is NULL. */
760  goto cleanup /* Goto the cleanup section. */
761 
762 error_handler:
763  /* If we got here there was an error, verify that it is so. */
764  g_assert (err != NULL)
765  /* Print error message. */
766  fprintf(stderr, "Error message: %s\n", err->message)
767  /* Make sure function status contains an error code. */
768  if (status == SUCCESS_CODE) status = err->code
769  /* Free the GError object. */
770  g_error_free(err)
771 
772 cleanup:
773  /* Free any allocated memory. */
774  if (some_vector) free(some_vector)
775 
776  ...
777 
778  /* Return program status. */
779  return status
780 }
781 
782 ...
783 
784 /* This function is GError-aware, and will initialize the GError */
785 /* object if an error occurs. The GError object usually comes */
786 /* as the last parameter. */
787 void some_function(some params, GError** err)
788 {
789 
790  ...
791 
792  FILE* fp
793  const char* filename = "somefile.txt"
794 
795  ...
796 
797  /* Try to open a file. This function is not GError aware. */
798  fp = fopen(filename, "r")
799 
800  /* Catch possible error in non-GError aware function. */
801  gef_if_error_create_goto(
802  *err, /* GError object. */
803  SOME_QUARK_ERROR, /* GLib GQuark identifier. */
804  fp == NULL, /* Error condition. */
805  SOME_ERROR_CODE, /* Error code to set in err. */
806  error_handler, /* Label to goto in case of error. */
807  "Unable to open file %s", /* Error msg to set in err. */
808  filename /* Extra args for error msg. */
809  )
810 
811  ...
812 
813  /* If we got here, everything is OK. */
814  /* It's good practice to check if err is NULL (caller doesn't */
815  /* care about error reporting OR if a non-null err is */
816  /* pointing to NULL (i.e. no error was reported). */
817  g_assert (err == NULL || *err == NULL)
818 
819  /* Goto finish label, ignoring the error handling section. */
820  goto finish
821 
822 error_handler:
823  /* If we got here there was an error, verify that it is so, */
824  /* i.e. either the caller doesn't care about error reporting, */
825  /* in which case err is NULL, OR a non-null err is in fact */
826  /* pointing to an initialized GError object. */
827  g_assert (err == NULL || *err != NULL)
828 
829  /* Run any other error handling code. */
830  ...
831 
832 finish:
833 
834  /* Close the file, if open. */
835  if (fp) fclose(fp)
836 
837  /* Perform additional required cleanup (free's and so on). */
838  ...
839 
840  /* Bye. */
841  return
842 
843 }
844 
845 ~~~~~~~~~~~~~~~
846 
847 As can be observed, GErrorF enforces a strict programming pattern,
848 which requires that complying functions follow a set of rules:
849 
850 * Define all pointers and objects in the beginning of the function
851 and initialize them to NULL.
852 * Define a pointer to a GError object and set it to NULL.
853 * Always test for errors after calls to error throwing functions
854 with `gef_if_error_create_goto` (non-GError aware functions) or
855 `gef_if_error_goto` (GError aware functions). If an error occurs
856 program execution will jump to the error handling block.
857 * Use the following pattern at the end of the function:
858  * goto cleanup
859  * error handling block
860  * cleanup: perform clean up
861  * return (possibly with a status code)
862 
863 This pattern avoids many bugs and makes error catching and handling
864 possible in C. However it is not to everyone's taste, and is thus
865 a completely optional aspect of _cf4ocl_.
