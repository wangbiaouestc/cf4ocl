4 ### Introduction
5 
6 This tutorial is based on the `canon` example available in the
7 examples folder. The goal is to add two vectors, `a` and `b`, as well as
8 a constant `d`, and save the result in a third vector, `c`. The OpenCL
9 kernel which performs this operation is given in the following code:
10 
11 ~~~~~~~~~~~~~~~{.c}
12 __kernel void sum(__global const uint *a, __global const uint *b,
13  __global uint * c, uint d, uint buf_size) {
14 
15  /* Get global ID. */
16  uint gid = get_global_id(0);
17 
18  /* Only perform sum if this workitem is within the size of the
19  * vector. */
20  if (gid < buf_size)
21  c[gid] = a[gid] + b[gid] + d;
22 }
23 ~~~~~~~~~~~~~~~
24 
25 For the purpose of this tutorial, we'll assume the kernel code is in a
26 file called `mysum.cl`, and the host code in `mysum.c`.
27 
28 ### Getting started
29 
30 The _cf4ocl_ header should be included at the beggining of the `mysum.c`
31 file:
32 
33 ~~~~~~~~~~~~~~~{.c}
34 #include <cf4ocl2.h>
35 ~~~~~~~~~~~~~~~
36 
37 The next step is to get an context with an OpenCL device where we can
38 perform our computation. _cf4ocl_ has several constructor functions for
39 creating contexts with different types of devices, some very simple,
40 some very flexible. For example, ::ccl_context_new_from_menu() lets
41 the user select the OpenCL device if more than one is available in the
42 system, and returns a context containing the selected device. Let's use
43 it:
44 
45 ~~~~~~~~~~~~~~~{.c}
46 
47 int main() {
48 
49  /* Variables. */
50  CCLContext * ctx = NULL;
51 
52  /* Code. */
53  ctx = ccl_context_new_from_menu(NULL);
54 
55 ~~~~~~~~~~~~~~~
56 
57 Where we pass `NULL` we could have passed an error management object,
58 which we'll discuss in detail further ahead. Error-throwing _cf4ocl_
59 signal errors in two ways: 1) using the return value; and, 2) populating
60 the error management object. In this case, because we're not passing
61 this object, we have to rely on the return value to check for errors. A
62 `NULL` return value indicates an error in all _cf4ocl_ constructors:
63 
64 ~~~~~~~~~~~~~~~{.c}
65  if (ctx == NULL) exit(-1);
66 ~~~~~~~~~~~~~~~
67 
68 In _cf4ocl_, all objects created with `new` constructors must be
69 release using the respective `destroy` destructors. For contexts, this
70 is the ::ccl_context_destroy() function:
71 
72 ~~~~~~~~~~~~~~~{.c}
73  /* Destroy context wrapper. */
74  ccl_context_destroy(ctx);
75 ~~~~~~~~~~~~~~~
76 
77 We now have compilable, leak-free _cf4ocl_ program, although it doesn't
78 do very much yet:
79 
80 ~~~~~~~~~~~~~~~{.c}
81 #include <cf4ocl2.h>
82 
83 int main() {
84 
85  /* Variables. */
86  CCLContext * ctx = NULL;
87 
88  /* Code. */
89  ctx = ccl_context_new_from_menu(NULL);
90  if (ctx == NULL) exit(-1);
91 
92  /* Destroy context wrapper. */
93  ccl_context_destroy(ctx);
94 
95  return 0;
96 }
97 ~~~~~~~~~~~~~~~
98 
99 We can compile the program with `gcc` (or `clang`), and run it:
100 
101 ~~~~~~~~~~~~~~~
102 $ gcc `pkg-config --cflags cf4ocl2` mysum.c -o mysum `pkg-config --libs cf4ocl2`
103 $ ./mysum
104 ~~~~~~~~~~~~~~~
105 
106 With Clang the command is the same, just replace `gcc` with `clang`.

108 ### Host and device buffers
109 
110 The goal of the program is to sum two vectors and a constant. Let's
111 declare two host vectors with some values, a third host vector which
112 will hold the result, three device buffers and the constant.
113 
114 ~~~~~~~~~~~~~~~{.c}
115 
116 #define VECSIZE 8
117 #define SUM_CONST 3
118 
119  //...
120 
121  /* Variables. */
122  CCLContext * ctx = NULL;
123  CCLBuffer * a = NULL, * b = NULL, * c = NULL;
124  cl_uint vec_a[VECSIZE] = {0, 1, 2, 3, 4, 5, 6, 7};
125  cl_uint vec_b[VECSIZE] = {3, 2, 1, 0, 1, 2, 3, 4};
126  cl_uint vec_c[VECSIZE];
127  cl_uint d = SUM_CONST;
128 
129 ~~~~~~~~~~~~~~~
130 
131 It's necessary to instantiate the device buffers, of which `a` and `b`
132 will be initialized with the values in the respective host vectors:
133 
134 ~~~~~~~~~~~~~~~{.c}
135 
136  //...
137 
138  /* Code. */
139  ctx = ccl_context_new_from_menu(NULL);
140  if (ctx == NULL) exit(-1);
141 
142  /* Instantiate and initialize device buffers. */
143  a = ccl_buffer_new(ctx, CL_MEM_READ_ONLY | CL_MEM_COPY_HOST_PTR,
144  VECSIZE * sizeof(cl_uint), vec_a, NULL);
145  if (a == NULL) exit(-1);
146 
147  b = ccl_buffer_new(ctx, CL_MEM_READ_ONLY | CL_MEM_COPY_HOST_PTR,
148  VECSIZE * sizeof(cl_uint), vec_b, NULL);
149  if (b == NULL) exit(-1);
150 
151  c = ccl_buffer_new(ctx, CL_MEM_WRITE_ONLY,
152  VECSIZE * sizeof(cl_uint), NULL, NULL);
153  if (c == NULL) exit(-1);
154 
155  //...
156 
157 ~~~~~~~~~~~~~~~
158 
159 Don't forget the destructors at the end:
160 
161 ~~~~~~~~~~~~~~~{.c}
162 
163  //...
164 
165  /* Destroy cf4ocl wrappers. */
166  ccl_buffer_destroy(c);
167  ccl_buffer_destroy(b);
168  ccl_buffer_destroy(a);
169  ccl_context_destroy(ctx);
170 
171  return 0;
172 
173 ~~~~~~~~~~~~~~~

221 ### Creating the command queue
222 
223 The ::ccl_queue_new() constructor provides the simplest way to create a
224 command queue. However, command queue creation requires a device. At
225 this time, only the context has the memory address of the selected
226 device. To get a pointer to the device, one can use the
227 ::ccl_context_get_device() function:
228 
229 ~~~~~~~~~~~~~~~{.c}
230  /* Variables. */
231  CCLContext * ctx = NULL;
232  CCLDevice* dev = NULL;
233  //...
234 
235  /* Get the selected device. */
236  dev = ccl_context_get_device(ctx, 0, NULL);
237  if (dev == NULL) exit(-1);
238 ~~~~~~~~~~~~~~~
239 
240 There's no need to release the device object. It will be automatically
241 released when the context is destroyed, in accordance to the
242 @ref ug_new_destroy "new/destroy" rule.
243 
244 Now we can create the command queue. We don't require any special
245 queue properties for now, so we pass `0` in the third argument:
246 
247 ~~~~~~~~~~~~~~~{.c}
248  /* Variables. */
249  CCLContext * ctx = NULL;
250  CCLDevice* dev = NULL;
251  CCLQueue* queue = NULL;
252  //...
253 
254  /* Create a command queue. */
255  queue = ccl_queue_new(ctx, dev, 0, NULL);
256  if (queue == NULL) exit(-1);
257  //...
258 
259  /* Destroy cf4ocl wrappers. */
260  //...
261  ccl_queue_destroy(queue);
262 ~~~~~~~~~~~~~~~
263 
264 Both ::ccl_context_get_device() and ::ccl_queue_new() expect an error
265 handling object as the last argument. By passing `NULL` we must rely
266 on the return value of these functions in order to check for errors.
267 Here's the complete program so far:

325 ### Creating and building the program
326 
327 _cf4ocl_ provides several constructors for creating
328 @ref CCL_PROGRAM_WRAPPER "program objects". When a single OpenCL C
329 kernel source file is involved, the most adequate constructor is
330 ::ccl_program_new_from_source_file():
331 
332 ~~~~~~~~~~~~~~~{.c}
333 
334  /* Variables. */
335  //...
336  CCLProgram* prg = NULL;
337 
338  //...
339  /* Create program. */
340  prg = ccl_program_new_from_source_file(ctx, "mysum.cl", NULL);
341  if (prg == NULL) exit(-1);
342  //...
343 
344  /* Destroy cf4ocl wrappers. */
345  ccl_program_destroy(prg);
346  //...
347 
348 ~~~~~~~~~~~~~~~
349 
350 Building the program is just as easy. For this purpose, we use the
351 ::ccl_program_build() function which returns `CL_TRUE` if the build is
352 successful or `CL_FALSE` otherwise:
353 
354 ~~~~~~~~~~~~~~~{.c}
355 
356  /* Variables. */
357  //...
358  cl_bool status;
359 
360  //...
361  /* Build program. */
362  status = ccl_program_build(prg, NULL, NULL);
363  if (!status) exit(-1);
364 
365 ~~~~~~~~~~~~~~~

448 ### Setting kernel arguments and running the program
449 
450 _cf4ocl_ greatly simplifies the execution of OpenCL programs. Instead
451 of creating a kernel with clCreateKernel(), setting kernel arguments
452 one-by-one with clSetKernelArg(), and finally executing the kernel with
453 clEnqueueNDRangeKernel(), _cf4ocl_ allows client code to do this using
454 a single function:
455 
456 ~~~~~~~~~~~~~~~{.c}
457 
458  /* Variables. */
459  //...
460  size_t gws = VECSIZE;
461  CCLEvent* evt = NULL;
462 
463  //...
464  /* Set kernel arguments and run kernel. */
465  evt = ccl_program_enqueue_kernel(prg, "sum", queue, 1, &gws,
466  NULL, NULL, NULL, a, b, c, ccl_arg_priv(d, cl_uint),
467  ccl_arg_priv(gws, cl_uint), NULL);
468  if (!evt) exit(-1);
469 
470 ~~~~~~~~~~~~~~~
471 
472 Buffer, image and sampler objects can be passed directly as kernel
473 arguments. Local and private variables are passed using the
474 ::ccl_arg_local() and ::ccl_arg_priv() macros, respectively.
475 
476 A local work size vector is expected as the 6th argument to
477 ::ccl_program_enqueue_kernel(). In this example, we pass `NULL`, which
478 means that the local work size is to be automatically determined by the
479 OpenCL implementation (as specified in the clEnqueueNDRangeKernel()
480 documentation). Often we need more control over this value, because
481 OpenCL implementations don't let us know what local work size was
482 effectively used. It can be a bit of a chore to determine a local work
483 size, especially when multiple dimensions are involved. Among other
484 things, it's necessary to check kernel and device limits. The
485 ::ccl_kernel_suggest_worksizes() is a very versatile function which can
486 help in this regard.
487 
488 While the ::ccl_program_enqueue_kernel() simplifies executing a kernel
489 (including setting its arguments), _cf4ocl_ provides additional
490 functions which allow client code to have finer control over this
491 process.

493 ### Checking results
494 
495 To check the results of the kernel execution, it's necessary to first
496 read the `c` output buffer from the device:
497 
498 ~~~~~~~~~~~~~~~{.c}
499 
500  /* Read the output buffer from the device. */
501  evt = ccl_buffer_enqueue_read(c, queue, CL_TRUE, 0,
502  VECSIZE * sizeof(cl_uint), vec_c, NULL, NULL);
503  if (!evt) exit(-1);
504 
505 ~~~~~~~~~~~~~~~
506 
507 Now we can check the results:
508 
509 ~~~~~~~~~~~~~~~{.c}
510 
511  /* Variables. */
512  //...
513  int i;
514  //...
515 
516  /* Check for errors. */
517  for (i = 0; i < VECSIZE; ++i) {
518  if (vec_c[i] != vec_a[i] + vec_b[i] + c) {
519  fprintf(stderr, "Unexpected results.\n");
520  exit(-1);
521  }
522  }
523  /* No errors found. */
524  printf("Results OK!\n");
525 
526 ~~~~~~~~~~~~~~~

623 ### A better way to check for errors
624 
625 Our program may be correctly implemented, but a number of OpenCL errors
626 can still occur. Checking the return values of _cf4ocl_ functions
627 allows us only to determine that something went wrong, not what went
628 wrong. Fortunately, all error-throwing _cf4ocl_ functions accept the
629 memory location of [GError](https://developer.gnome.org/glib/stable/glib-Error-Reporting.html#GError)
630 error handling object, usually the last function argument. If given,
631 this object will be populated with an error code and an error domain, as
632 well as an error message, if an error occurs. In our program we're
633 passing `NULL` where the `GError` object memory location was expected,
634 so no information about errors is made available to the caller. To
635 change the way we're handling errors, we must first declare a `GError`
636 object, and initialize it to `NULL`:
637 
638 ~~~~~~~~~~~~~~~{.c}
639 
640  /* Variables. */
641  //...
642  GError * err = NULL;
643 ~~~~~~~~~~~~~~~
644 
645 Then, we should pass the memory location of this object to all _cf4ocl_
646 error-throwing functions, e.g.:
647 
648 ~~~~~~~~~~~~~~~{.c}
649  /* Create context with user selected device. */
650  ctx = ccl_context_new_from_menu(&err);
651  //...
652 
653  /* Get the selected device. */
654  dev = ccl_context_get_device(ctx, 0, &err);
655  //...
656 
657  /* Create a command queue. */
658  queue = ccl_queue_new(ctx, dev, 0, &err);
659  //...
660 ~~~~~~~~~~~~~~~
661 
662 Now we can check this object after _cf4ocl_ function calls. Let's create
663 a small macro to do so:
664 
665 ~~~~~~~~~~~~~~~{.c}
666 #define CHECK_ERROR(err) \
667  if (err != NULL) { fprintf(stderr, "\n%s\n", err->message); exit(-1); }
668 ~~~~~~~~~~~~~~~
669 
670 We can also remove the `status` and `evt` variables, because we don't
671 rely on them anymore for error checking. Here's the complete program,
672 with more informative error checking:
673 
674 ~~~~~~~~~~~~~~~{.c}
675 #include <cf4ocl2.h>
676 
677 #define VECSIZE 8
678 #define SUM_CONST 3
679 #define CHECK_ERROR(err) \
680  if (err != NULL) { fprintf(stderr, "\n%s\n", err->message); exit(-1); }
681 
682 int main() {
683 
684  /* Variables. */
685  CCLContext * ctx = NULL;
686  CCLDevice * dev = NULL;
687  CCLQueue * queue = NULL;
688  CCLProgram * prg = NULL;
689  CCLBuffer * a = NULL, * b = NULL, * c = NULL;
690  cl_uint vec_a[VECSIZE] = {0, 1, 2, 3, 4, 5, 6, 7};
691  cl_uint vec_b[VECSIZE] = {3, 2, 1, 0, 1, 2, 3, 4};
692  cl_uint vec_c[VECSIZE];
693  cl_uint d = SUM_CONST;
694  size_t gws = VECSIZE;
695  int i;
696  GError * err = NULL;
697 
698  /* Create context with user selected device. */
699  ctx = ccl_context_new_from_menu(&err);
700  CHECK_ERROR(err);
701 
702  /* Get the selected device. */
703  dev = ccl_context_get_device(ctx, 0, &err);
704  CHECK_ERROR(err);
705 
706  /* Create a command queue. */
707  queue = ccl_queue_new(ctx, dev, 0, &err);
708  CHECK_ERROR(err);
709 
710  /* Instantiate and initialize device buffers. */
711  a = ccl_buffer_new(ctx, CL_MEM_READ_ONLY | CL_MEM_COPY_HOST_PTR,
712  VECSIZE * sizeof(cl_uint), vec_a, &err);
713  CHECK_ERROR(err);
714 
715  b = ccl_buffer_new(ctx, CL_MEM_READ_ONLY | CL_MEM_COPY_HOST_PTR,
716  VECSIZE * sizeof(cl_uint), vec_b, &err);
717  CHECK_ERROR(err);
718 
719  c = ccl_buffer_new(ctx, CL_MEM_WRITE_ONLY,
720  VECSIZE * sizeof(cl_uint), NULL, &err);
721  CHECK_ERROR(err);
722 
723  /* Create program. */

725  CHECK_ERROR(err);
726 
727  /* Build program. */
728  ccl_program_build(prg, NULL, &err);
729  CHECK_ERROR(err);
730

732  NULL, NULL, &err, a, b, c, ccl_arg_priv(d, cl_uint),
733  ccl_arg_priv(gws, cl_uint), NULL);
734  CHECK_ERROR(err);
735 
736  /* Read the output buffer from the device. */
737  ccl_buffer_enqueue_read(c, queue, CL_TRUE, 0,
738  VECSIZE * sizeof(cl_uint), vec_c, NULL, &err);
739  CHECK_ERROR(err);
740 
741  /* Some OpenCL implementations don't respect the blocking read,
742  * so this guarantees that the read is effectively finished. */
743  ccl_queue_finish(queue, &err);
744  CHECK_ERROR(err);
745 
746  /* Check for errors. */
747  for (i = 0; i < VECSIZE; ++i) {
748  if (vec_c[i] != vec_a[i] + vec_b[i] + d) {
749  fprintf(stderr, "Unexpected results.\n");
750  exit(-1);
751  }
752  }
753  /* No errors found. */
754  printf("Results OK!\n");
755 
756  /* Destroy cf4ocl wrappers. */
757  ccl_program_destroy(prg);
758  ccl_buffer_destroy(c);
759  ccl_buffer_destroy(b);
760  ccl_buffer_destroy(a);
761  ccl_queue_destroy(queue);
762  ccl_context_destroy(ctx);
763 
764  return 0;
765 }

772 ### Profiling your program the easy way
773 
774 Profiling OpenCL code by hand, i.e. by gathering and processing
775 information about all `cl_event` objects, is an extremely verbose and
776 error-prone process. However, it comes for free with _cf4ocl_. Well,
777 mostly. The first change we must perform on our program is to enable
778 profiling when the command queue is created:
779 
780 ~~~~~~~~~~~~~~~{.c}
781 
782  /* Create a command queue. */
783  queue = ccl_queue_new(ctx, dev, CL_QUEUE_PROFILING_ENABLE, &err);
784 
785 ~~~~~~~~~~~~~~~
786 
787 Additionally, we copied host vectors to device implicitly during device
788 buffer creation. These implicit transfers don't produce OpenCL events,
789 and are thus unavailable for profiling. As such, we must make these
790 operations explicit:
791 
792 ~~~~~~~~~~~~~~~{.c}
793 
794  /* Instantiate device buffers. */
795  a = ccl_buffer_new(ctx, CL_MEM_READ_ONLY,
796  VECSIZE * sizeof(cl_uint), NULL, &err);
797  CHECK_ERROR(err);
798 
799  b = ccl_buffer_new(ctx, CL_MEM_READ_ONLY,
800  VECSIZE * sizeof(cl_uint), NULL, &err);
801  CHECK_ERROR(err);
802 
803  //...
804 
805  /* Initialize device buffers. */
806  ccl_buffer_enqueue_write(a, queue, CL_TRUE, 0,
807  VECSIZE * sizeof(cl_uint), vec_a, NULL, &err);
808  CHECK_ERROR(err);
809 
810  ccl_buffer_enqueue_write(b, queue, CL_TRUE, 0,
811  VECSIZE * sizeof(cl_uint), vec_b, NULL, &err);
812  CHECK_ERROR(err);
813 
814 ~~~~~~~~~~~~~~~
815 
816 We can now profile our program using functionality provided by the
817 @ref CCL_PROFILER "profiler module":
818 
819 ~~~~~~~~~~~~~~~{.c}
820 
821  /* Variables. */
822  //...
823  CCLProf* prof = NULL;
824 
825  //...
826 
827  /* Perform profiling. */
828  prof = ccl_prof_new();
829  ccl_prof_add_queue(prof, "queue1", queue);
830  ccl_prof_calc(prof, &err);
831  CHECK_ERROR(err);
832 
833  /* Show profiling info. */
834  ccl_prof_print_summary(prof);
835 
836  /* Destroy profiler object. */
837  ccl_prof_destroy(prof);
838 
839 ~~~~~~~~~~~~~~~
